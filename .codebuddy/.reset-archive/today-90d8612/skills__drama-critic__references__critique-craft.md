# 评审手艺（critique-craft）

> 这是 Critic 的"怎么看一场戏"能力册。
> Critic 不是查错员——Critic 是**戏剧顾问**。查错是底线，评审戏的好坏才是价值。
>
> 本册让 Critic 从"跑完 check-ai-taste 然后填 6 维度分数"升级到"**看出这场戏缺什么**"。

---

## 两种 Critic 模式

### 模式 A · 规则 Critic（底线）

这是 `check-ai-taste.js` 脚本能做的：查破折号、查独占段、查 C5/C6 黑名单。

**优点**：客观、可复现、零成本
**缺点**：只能查"有没有踩雷"，无法评价"戏好不好看"

### 模式 B · 戏剧 Critic（本册的目标）

人类 Critic 读完一场戏，能说出：
- "第 3 回合林墨的自嘲给得太早了，读者还没感受到压抑"
- "场景 4 陈队长录像的处理缺少'第三方目击'层，张教授应该有一个反应"
- "母亲的沉默应该更长一拍，你给的那个'停了一秒'太短"

这些**戏剧感的判断**是 check 脚本做不到的。

Critic Agent 的价值就在这里。本册就是教会 Critic 怎么做这种判断。

---

## 评审的五个层

看一场戏必须从上到下过 5 层。任何一层出问题都不能放过。

### 层 1 · 戏眼层

**核心问题**：这场戏的"戏眼"是什么？它被演出来了吗？

戏眼 = 场景里**那个一切为之存在**的瞬间（见 director-intuition.md 第 1 问"门"的概念）。

**典型戏眼**：
- 母亲叫"糕"少一个字（褪色第一次对至亲生效）
- 陈队长敲到第四下停，林墨也敲到第四下停（身体先答应了）
- 夕阳下沈砚之影子淡一度（沈砚之也在褪色的种子）

**Critic 评审要问**：
- 戏眼的**物理发生**有没有被清楚呈现？（读者能看见"糕糕→糕"的字数差）
- 戏眼前的**铺垫密度**是否足够？（前面有没有建立"以前是糕糕"的期待）
- 戏眼后的**余韵**是否到位？（林墨的沉默反应是不是让这一下的戏生效了）

戏眼没演出来 → 整场戏白给 → Critic Error 级

### 层 2 · 角色语言层

**核心问题**：每个角色的语言指纹兑现了吗？

参考 SOUL.yaml 的 `performance.voice.quirks`、`unique_markers`、`body_signature`、`silence_quota`、`breakdown_budget` 字段。

**Critic 评审要做**：
- 列出场景里每个角色的语言指纹预期（来自 SOUL）
- 对比 novel.md 里他的实际台词/动作
- 标出**未兑现**的指纹（例：林墨的"紧张时摸后颈"本场景没出现）
- 标出**错误兑现**的指纹（例：别的角色说"没关系"污染了林墨的口头禅）

### 层 3 · 信息合规层

**核心问题**：角色是不是只用了自己应该知道的信息？

参考 SOUL.yaml 的 `runtime.known_facts` 和 `active_secrets`（state.json）。

**典型违规**：
- 林墨在场景 4 谈论周伯的孙女（他还不知道这件事）
- 沈砚之当面告诉林墨"我也在褪色"（卷一合约期内他的 secret 不能泄）
- 某角色援引 MEMORY.md 中不存在的往事（凭空编造）

**Critic 评审要做**：
- 扫描每个角色的对白和内心
- 对照他的 known_facts + MEMORY → 超范围的内容标为 Error

### 层 4 · 戏剧张力层

**核心问题**：这场戏有没有张力？张力从哪里来？

戏剧张力的四个来源（导演书常说的）：
- **悬念**：读者不知道接下来发生什么
- **期待**：读者以为 A 会发生，看戏等它发生（或反转）
- **共情**：读者希望角色得到/避免什么
- **信息差**：读者知道的和角色知道的不同步（戏剧反讽）

**Critic 评审要做**：
- 标出本场戏张力的**主导来源**（四选一或混合）
- 评估张力**密度曲线**：开场-发展-戏眼-余韵，各阶段张力水平
- 标出**张力塌陷点**（读者会走神的段落）

### 层 5 · 文本执行层

**核心问题**：上面 4 层都对，但字面文本水准如何？

这一层就是 check-ai-taste 负责的：
- 破折号 / 独占段 / 三连排比
- C5 黑名单句式
- C6 第三人称叙事
- 破防戏五铁律
- 悬疑三铁律

**Critic 评审要做**：
- 跑脚本，拿 exit code
- 把脚本的 Warning/Error 写进报告作为"文本执行问题"

## 评审的工作流

### 阶段 1 · 输入收集

- novel.md（正文）
- beat-sheet.md（预期戏眼、节奏、钩子）
- interactions.jsonl（原始素材，Critic 能看到 [inner_thought]）
- 参演角色的 SOUL.yaml（语言指纹）
- world/state.json（信息合规边界）
- world/hooks-ledger.md（钩子状态）
- world/imagery-ledger.md（意象状态）

### 阶段 2 · 跑脚本

```bash
node .codebuddy/skills/drama-critic/scripts/check-ai-taste.js --story X --episode Y
```

拿到脚本结果作为"文本执行层"输入。

### 阶段 3 · 人工/Agent 评审五层

按上面的"五个层"从上到下过一遍。**不要倒过来过**——如果戏眼都没演出来，底下的规则全通过也没用。

### 阶段 4 · 产出 6 维度雷达

```
| 维度 | 权重 | 得分 |
|---|---|---|
| 人格一致性 | 20% | ... |
| 创伤响应 | 15% | ... |
| 语言保真度 | 15% | ... |
| 内心与外在张力 | 10% | ... |
| 秘密保护 | 10% | ... |
| 读者吸引力 | 25% | ... |
```

但**权重之外**要独立写出：
- **戏眼评语**（戏眼是什么，有没有演出来）
- **声音评语**（各角色声音是否区分度高）
- **张力评语**（张力曲线是否健康）
- **反模式标注**（是否踩了 AP-1 到 AP-6 任何一条，见 samples-and-antipatterns.md）

## 戏剧感判断的 10 个常见问题

Critic 读完一场戏应该能回答这 10 个问题。答不上来 → 评审不深入。

### Q1 · 这场戏的戏眼是什么？它被演出来了吗？
参见"层 1 · 戏眼层"。

### Q2 · 主视角在这场戏里被改变了吗？
戏的定义：主角的认知、关系、处境**之前** ≠ **之后**。
如果一场戏主角结束时和开始时一样 → 这场戏没必要存在。

### Q3 · 谁是"带着门进来的人"？他的目的达成了吗？
参见 director-intuition.md 第 2 问。隐藏主角的目的性是否清晰。

### Q4 · 谁在"关门"？他的抵抗有没有被呈现？
参见 director-intuition.md 第 3 问。反作用力的可见性。

### Q5 · 沉默者的物理存在感如何？
他们是透明人还是戏的证词？

### Q6 · 留白在哪一拍？留白够不够？
如果一场戏从头到尾满密度 → 读者窒息 → 需要标记补留白。

### Q7 · 场景结束时，哪些物品改变了位置？
物品的移动轨迹是戏的物理证据。

### Q8 · 读者在这场戏里学到了什么新信息？
一场戏必须给读者**新信息**（可以是事实、可以是角色关系、可以是世界观）。
没有新信息的戏 = 水戏。

### Q9 · 场景的"戏剧反讽"在哪里？
（读者知道 X，角色不知道 X）—— 几乎每场好戏都有，缺了则扁平。

### Q10 · 读完这场戏，读者想翻下一页吗？为什么？
**翻页欲**是本集读者吸引力的核心指标。

## 给 Critic Agent 的任务模板

当 Director 在 Phase 4 spawn 独立 Critic Agent 时，prompt 应含：

```
你是独立 Critic。请先读以下 references：
- .codebuddy/skills/drama-critic/references/critique-craft.md（本文）

然后按以下步骤评审 <story> 的 <episode>：

1. 跑脚本：
   node .codebuddy/skills/drama-critic/scripts/check-ai-taste.js --story X --episode Y
   
2. 读取素材：
   - novel.md
   - beat-sheet.md
   - interactions.jsonl（如果存在）
   - 参演角色 SOUL.yaml
   - world/state.json / hooks-ledger.md / imagery-ledger.md

3. 按五层评审：
   - 层 1 戏眼层：戏眼是什么？演出来了吗？
   - 层 2 角色语言层：语言指纹兑现？
   - 层 3 信息合规层：known_facts 边界？
   - 层 4 戏剧张力层：张力曲线？
   - 层 5 文本执行层：脚本结果。

4. 回答十问（不是每个都写长，但每个都要回答）。

5. 产出 critic-report.md，含：
   - 6 维度雷达评分
   - 五层评审结论
   - 十问回答
   - 反模式标注（对照 samples-and-antipatterns.md）
   - Error/Warning/Info 三级问题清单
   - 最高优先级修订建议（唯一一条，如果只能改一个地方改哪里）

你不是在"给一个作业打分"，你是在"把这场戏变得更好"。
```

---

## 最后一条原则

> 规则 Critic 告诉 Director "**你错了**"。
> 戏剧 Critic 告诉 Director "**这里本来可以更好**"。
>
> 前者是警察。后者是同行。
> Critic Agent 的使命是后者。
