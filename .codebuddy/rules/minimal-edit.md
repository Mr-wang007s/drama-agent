### 最小改动规则（反膨胀）

默认模式。防 AI 生成内容膨胀、防顺手重构、防过度抽象。

---

#### 触发边界

- **默认 = 最小修复模式**：只改用户明确要求的内容，不顺手改周边。
- **大改解锁**：用户消息显式出现"重构 / refactor / 重写 / 重新设计 / 进入 plan / 做个方案"等词，或走 EnterPlanMode 并被批准，才允许跨文件大改。
- 不确定时按最小模式执行，改完在回复末尾一句话问"要不要顺带做 X"，不擅自扩大。

---

#### 量化硬线（单次改动）

| 维度 | 上限 | 超出处理 |
|---|---:|---|
| 触及文件数 | ≤ 3 | 先列清单让用户确认 |
| 新增/修改行数（不含纯删除）| ≤ 50 | 先列清单让用户确认 |
| 新建文件 | ≤ 1 | 超过必须确认 |

纯删除、配置微调、文档 1-2 行小补丁不计入门控。

---

#### 执行原则

1. **先读后改**：修 bug 先定位根因所在行，再动刀；禁止"顺带优化"未读代码。
2. **同类优先**：有多条等价方案时，优先选改动面最小的那条（哪怕不是最优雅）。
3. **对比 2 版再落笔**：心里至少过一遍"这样改 vs 最小还能再小吗"，写出更小的那版。
4. **禁止顺手做**：重命名、提取函数、加注释、补类型、统一风格、删"看起来没用"的代码——除非用户要求。
5. **禁止防御式膨胀**：不加未要求的错误处理 / 兜底 / 校验 / 日志 / feature flag。信任内部调用方。
6. **改事实源优先**：规则 / SKILL.md / craft 是事实源；CODEBUDDY.md / README 是索引。改索引前先确认事实源已改。
7. **索引层变更不复述**：更新 CODEBUDDY / README 时只改受影响的那一行，不重写整节。

---

#### 自检清单（落笔前回顾）

```
□ 用户是否显式要求了"重构"？否 → 走最小模式
□ 触及文件 ≤ 3？新增行 ≤ 50？否 → 停下确认
□ 有没有顺手改了没要求的东西？有 → 撤掉
□ 有没有为"万一"加的代码？有 → 撤掉
□ 是否对比过更小的替代方案？没有 → 再想一版
```

---

#### 例外

用户使用以下表述时，本规则不适用：
- "重构" / "refactor" / "重写" / "重新设计"
- "做个方案" / "规划一下" / "进入 plan"
- EnterPlanMode 被批准后的 Plan 实施阶段
- 显式说"不用最小化" / "放开做"
